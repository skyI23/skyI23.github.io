基本概念

java 版本

JDK 1.x 阶段

  • JDK 1.0 到 JDK 1.4：
    • Java 平台最早的版本使用 JDK 1.x 的命名方式。例如：
      • JDK 1.0（1996年发布）：Java 的第一个正式版本。
      • JDK 1.1（1997年发布）：添加了许多改进，如内部类和 JDBC。
      • JDK 1.2（1998年发布）：引入了“Java 2”品牌，开始了 Java 2 平台，但内部版本号仍然是 1.2。
      • JDK 1.3（2000年发布）和 JDK 1.4（2002年发布）继续沿用 JDK 1.x 的命名方式。

Java x 阶段

  • Java 5（JDK 1.5）和 Java 6（JDK 1.6）：

    • 从 JDK 1.5 开始，Sun Microsystems（当时 Java 的维护者）决定简化命名方式，官方开始使用 Java x 的称呼来代替冗长的 JDK 1.x。这也是为了市场推广更简洁易记。例如：
      • JDK 1.5 被市场化命名为 Java 5。
      • JDK 1.6 被命名为 Java 6，但内部版本号依然是 1.6。

  • Java 7（JDK 1.7）和 Java 8（JDK 1.8）：

    • 这个模式继续应用于后续版本：
      • JDK 1.7 被称为 Java 7。
      • JDK 1.8 被称为 Java 8。

在这段时期，虽然官方文档和开发者社区中普遍使用 Java x 作为主要版本名称，但 JDK 的内部版本号仍然保持 1.x 格式。例如，Java 8 仍然是 1.8.0_xx 的格式。

JDK x 阶段

  • JDK 9 及之后：

    • 从 JDK 9 开始，Oracle 决定完全放弃 1.x 的内部版本号格式。为了简化版本管理，JDK 9 及后续版本直接使用主版本号作为版本名称：

      • JDK 9（2017年发布）：正式放弃 1.x 的版本号，直接命名为 JDK 9。
      • JDK 10、JDK 11（LTS，2018年发布）等后续版本继续采用这个命名方式。

    • 这意味着从 JDK 9 开始，版本号变得更加统一和简洁，不再存在内部版本号和市场名称不一致的情况。

  • Method：表示类的方法，可以调用这些方法。
  • Field：表示类的字段，可以读取和修改字段的值。
  • Constructor：表示类的构造函数，可以创建类的实例。

代理

在 Java 编程中，代理模式是一种设计模式，它允许开发者在方法调用的前后插入额外的逻辑。这种技术在许多实际应用中非常有用，例如日志记录、事务管理和安全检查。Java 代理模式可以分为静态代理和动态代理。

静态代理

静态代理的概念

静态代理是指在编译时由开发者创建的代理类。代理类与被代理类实现相同的接口，并在代理类中调用被代理类的方法。代理类可以在调用方法前后添加额外的逻辑。

静态代理的优缺点

  • 优点：

    • 实现简单，逻辑清晰。
    • 可以完全控制代理类的行为。

  • 缺点：

    • 每个接口都需要手动编写代理类，代码冗余较多。
    • 不灵活，无法在运行时动态改变代理行为。

动态代理

动态代理的概念

动态代理指的是在运行时动态生成代理类，而不是在编译时手动编写。Java 提供了基于接口的动态代理和基于类的动态代理（通过第三方库实现，如 CGLIB）。

  • 基于接口的动态代理：使用 JDK 提供的 java.lang.reflect.Proxy 类和 InvocationHandler 接口。
  • 基于类的动态代理：通常使用 CGLIB 或者其他字节码操作库实现。

基于接口的动态代理

  • InvocationHandler 接口：需要实现 invoke 方法，该方法在代理对象调用任何方法时都会被执行。
  • Proxy.newProxyInstance 方法：用于创建代理对象，动态生成实现指定接口的代理类，并使用 InvocationHandler 来处理方法调用。

基于类的动态代理（CGLIB）

当目标类没有实现接口时，可以使用 CGLIB 来创建代理。

类加载机制

Java 的类加载机制是 JVM（Java Virtual Machine）中一个非常重要的部分，它负责将类的字节码加载到内存中，并将其转换为可以被 JVM 执行的类对象。

Java 虚拟机规范并没有指明二进制字节流要从一个 Class 文件获取，或者说根本没有指明从哪里获取、怎样获取。这种开放性使得 Java 在很多领域得到充分运用，例如：

  • 从ZIP包中读取：这很常见，成为 JAR，EAR，WAR 格式的基础。
  • 从网络中获取：最典型的应用就是 Applet。
  • 运行时计算生成：最典型的是动态代理技术，在 java.lang.reflect.Proxy 中，就是用了 ProxyGenerator.generateProxyClass 来为特定接口生成形式为 $Proxy 的代理类的二进制字节流。
  • 由其他文件生成：最典型的 JSP 应用，由 JSP 文件生成对应的 Class 类。

类加载的过程

类的加载过程包括以下几个阶段：

加载（Loading）

  • 通过类的全限定名获从文件系统、JAR 包、网络等不同的源查找类的字节码。
  • 将字节码代表的静态存储结构转化为方法区的运行时数据结构。
  • 在内存中生成一个代表该类的 java.lang.Class 对象，作为方法区这个类的各种数据的访问入口。

链接（Linking）

  • 验证（Verification）：检查类文件的字节码是否合法且符合 JVM 的规范，确保字节码不会破坏 JVM 的安全性。
  • 准备（Preparation）：为类的静态变量分配内存，并将其初始化为默认值（如 0、null、false 等）。
  • 解析（Resolution）：将常量池中的符号引用（如类名、方法名、字段名）替换为直接引用（即实际的内存地址）。这一步是懒惰的，即在实际使用时才解析。

初始化（Initialization）

  • 初始化是类加载的最后一步。JVM 执行类的静态初始化块（static {}）和静态变量的初始化代码。
  • 初始化阶段是类加载过程的真正开始执行 Java 程序代码的时刻。静态变量的赋值操作和静态块的执行就是在这一阶段进行的。

类加载方式

显式类加载（Explicit Class Loading）

显式类加载，也称为类的动态加载，通过编程的方式在运行时加载类对象。主要方法包括：

  • 反射机制：通过 Class.forName 方法加载类。会执行类的初始化。

  • 类加载器：通过 ClassLoader 的 loadClass 方法加载类。

隐式类加载（Implicit Class Loading）

隐式类加载是在程序编写阶段，编译器根据代码自动决定何时加载类。例如：

  • new 类实例：创建类的实例。

  • **类名.方法名()**：通过类名直接调用静态方法。

类加载器

Java的类加载器（Class Loader）是Java虚拟机（JVM）的一部分，负责在运行时动态加载类。类加载器机制允许Java程序通过动态链接加载类，实现模块化和动态扩展。

类加载器的类型

在 JVM 类加载器主要有三种：

  • Bootstrap ClassLoader（引导类加载器）：
    • 负责加载核心 Java 类库，如 rt.jar。
    • 由 JVM 内部实现，使用本地代码（通常是 C/C++）编写。
    • 不可见于 Java 层，无法直接通过 Java 代码访问。
  • Extension ClassLoader（扩展类加载器）：
    • 加载扩展类库，通常位于 jre/lib/ext 目录中。
    • 继承自 java.lang.ClassLoader。
  • Application ClassLoader（应用/系统类加载器）：
    • 加载应用程序类路径（CLASSPATH）中的类。
    • 是系统默认的类加载器，用于加载用户自定义的类和第三方库。

而从 Java 虚拟机的角度来说，只存在两种不同的类加载器：

  • 一种是启动类加载器（Bootstrap ClassLoader），这个类加载器使用 C++ 语言实现（HotSpot 虚拟机中），是虚拟机自身的一部分；
  • 另一种就是所有其他的类加载器，这些类加载器都有 Java 语言实现，独立于虚拟机外部，并且全部继承自 java.lang.ClassLoader。

AppClassLoader 是默认的类加载器，如果类加载时我们不指定类加载器的情况下，默认会使用 AppClassLoader 加载类，ClassLoader.getSystemClassLoader() 返回的系统类加载器也是 AppClassLoader。

某些时候我们获取一个类的类加载器时候可能会返回一个 null 值，如：java.io.File.class.getClassLoader() 将返回一个 null 对象，因为 java.io.File 类在 JVM 初始化的时候会被 Bootstrap ClassLoader（引导类加载器）加载（该类加载器实现于 JVM 层，采用 C++ 编写），我们在尝试获取被 Bootstrap ClassLoader 类加载器所加载的类的 ClassLoader 时候都会返回 null。

类加载器与类的唯一性

类加载器虽然只用于实现类的加载动作，但是对于任意一个类，都需要由加载它的类加载器和这个类本身共同确立其在 Java 虚拟机中的唯一性。通俗的说，JVM 中两个类是否“相等”，首先就必须是同一个类加载器加载的，否则，即使这两个类来源于同一个 Class 文件，被同一个虚拟机加载，只要类加载器不同，那么这两个类必定是不相等的。

这里的“相等”，包括代表类的 Class 对象的 equals() 方法、isAssignableFrom() 方法、isInstance() 方法的返回结果，也包括使用 instanceof 关键字做对象所属关系判定等情况。

类加载器常见方法

  • loadClass（加载指定的 Java 类）

    加载名称为 name 的类。如果类已经被加载，则直接返回该类的 Class 对象。如果没有加载过，则按双亲委派模型先委派给父类加载器，若父类加载器无法加载，再调用 findClass 方法来尝试加载。

  • findClass（查找指定的 Java 类）

    根据类的名称查找并加载类的字节码。通常由自定义类加载器实现，用于从特定的来源加载类。

  • findLoadedClass（查找 JVM 已经加载过的类）

    查找 JVM 已经加载过的类。如果类已经加载，则返回该类的 Class 对象；否则返回 null。

  • defineClass（定义一个 Java 类）

    将字节数组转换为一个 Class 对象。这个方法是 ClassLoader 的核心，用于将字节码定义为类。

    • **String name**：要定义的类的二进制名称（Binary Name），即类的全限定名（例如，com.example.MyClass）。如果此参数为 null，则表示该类没有名称。通常建议提供类名以便于调试和管理。
    • **byte[] b**：包含类的字节码的字节数组。这个字节数组包含了通过编译器或其他方式生成的 Java 字节码，通常是 .class 文件的内容。
    • **int off**：字节数组 b 中，字节码数据的起始偏移量。表示从数组 b 的哪个位置开始读取数据。一般情况下，如果整个数组都包含字节码数据，这个值为 0。
    • **int len**：要从字节数组 b 中读取的字节码数据的长度。表示从偏移量 off 开始，读取多长的数据来定义类。通常这个值是 b.length - off。

  • resolveClass（链接指定的 Java 类）

    链接指定的 Java 类。这个方法会完成类的验证、准备和解析步骤，使得类在 JVM 中可以被正确使用。

双亲委派模型

双亲委派模型（Parent Delegation Model）是 Java 类加载器机制中的一种设计模式，其主要目的是确保 Java 核心类库的安全性和稳定性。通过这种模型，类加载器在加载类时，会先将请求委托给父类加载器处理，如果父类加载器无法加载这个类，才由当前类加载器尝试加载。

这种机制保证了Java核心类库（如java.lang.*、java.util.*等）始终由引导类加载器（Bootstrap ClassLoader）加载，从而避免了类的重复加载和冲突，确保了 Java 核心类库的安全性。

类加载器层次结构

除了前面提到的三种类加载器外，还有自定义的类加载器，它们之间的层次关系被称为类加载器的双亲委派模型。该模型要求除了顶层的启动类加载器外，其余的类加载器都应该有自己的父类加载器，而这种父子关系一般通过组合（Composition）关系来实现，而不是通过继承（Inheritance）。

双亲委派模型具体实现

在Java中，每个类加载器都有一个父类加载器，类加载器在加载类时，遵循以下步骤：

  1. 检查当前类是否已经加载：如果已经加载，则直接返回。
  2. 委托父类加载器加载：将加载请求委托给父类加载器，依次递归，直到最顶层的Bootstrap ClassLoader。
  3. 父类加载器无法加载：如果父类加载器无法加载该类，则当前类加载器尝试自己加载。

自定义类加载器

基本概念

在Java中，实现自定义类加载器通常是通过继承 java.lang.ClassLoader 类并重写其 findClass() 方法来实现的。自定义类加载器允许我们定义从哪里以及如何加载类，并提供了扩展和增强 Java 类加载行为的能力。

java.lang.ClassLoader 类的基本职责是根据指定的类名称，找到或生成其对应的字节码，然后定义出一个 Java 类（即 java.lang.Class 类的实例）。除此之外，ClassLoader 还负责加载 Java 应用所需的资源，如图像文件和配置文件等。

双亲委派模型相关

双亲委派模型在 JDK 1.2 引入，是一种推荐的类加载模型。其核心思想是：一个类加载器在加载类时，首先将请求委派给父类加载器，只有当父类加载器无法完成加载请求时，才尝试自己加载。这个模型确保了 Java 核心类库的安全性和统一性。

在 JDK 1.2 之前，自定义类加载器通常需要重写 loadClass 方法来实现类加载功能，但从 JDK 1.2 开始，引入了双亲委派模型，开发者通常只需要重写 findClass 方法即可（loadClass 已经被双亲委派模型实现好了）。

不遵循双亲委派模型的情况：

  • JDK 1.2之前的类加载器：需要覆盖 loadClass 方法来实现类加载功能，没有遵循双亲委派模型。
  • 线程上下文类加载器：为了实现基础类调用下层代码，可以通过 Thread.setContextClassLoader 设置线程上下文类加载器，不遵循双亲委派模型。
  • OSGi技术：为了实现模块热替换和热部署，OSGi 类加载器体系发展为网状结构，没有完全遵循双亲委派模型。

defineClass

不管是加载远程 class 文件，还是本地的 class 或 jar 文件，Java 都经历的是 loadClass、findClass、defineClass 这三个方法调用。

TemplatesImpl

TemplatesImpl 是 Java 中存在于 Xalan XSLT 处理器中的一个类，它实现了 javax.xml.transform.Templates 接口。TemplatesImpl 是 XSLT 处理中的一个重要类，主要用于表示一个经过编译的 XSLT 样式表，并且可以通过这个类多次应用样式表来转换 XML 文档。

TemplatesImpl 类内部有几个关键的字段和方法，这些字段在安全研究中经常被提及：

  • **private String _name**：样式表的名称。
  • **private byte[][] _bytecodes**：存储了经过编译的字节码的数组，这些字节码会在运行时被加载并执行。
  • **private transient TransformerFactoryImpl _tfactory**：TransformerFactory 的引用，用于生成 Transformer 对象。

这些字段中的 _bytecodes 和 _class 是特别重要的，因为它们直接涉及到字节码的加载和类的定义。

com.sun.org.apache.xalan.internal.xsltc.trax.TemplatesImpl 类中定义的内部类 TransletClassLoader 是一个自定义的类加载器。这个类加载器的主要作用是用于加载和定义通过 XSLT 编译的 Java 类（称为 translet）。这些类是在 XSLT 编译过程中动态生成的，用于执行特定的 XSLT 转换操作。

这个类里重写了 defineClass 方法，并且这里没有显式地声明其定义域。Java中默情况下，如果一个方法没有显式声明作用域，其作用域为 default。所以也就是说这里的 defineClass 由其父类的 protected 类型变成了一个 default 类型的方法，可以被类外部调用。
